● Here are the two restore workflows we used:                                                                                         
                                                                                                                                      
  1. virsh save / virsh restore (managed save)                                                                                        
                                                                                                                                      
  Save (stops the VM, writes memory to file):                                                                                         
  sudo virsh save l1-ubuntu /root/nested_qemu/test.save --bypass-cache
                                                                                                                                      
  Restore (resumes from saved memory; disk is whatever it is now):                                                                    
  sudo virsh restore /root/nested_qemu/test.save --bypass-cache                                                                       
                                                                                                                                      
  2. External snapshot revert (manual — libvirt doesn't auto-revert externals)                                                        
                                                                                                                                      
  Create snapshot (live, disk + memory):                                                                                              
  sudo virsh snapshot-create-as l1-ubuntu --name snap1 --memspec file=/root/nested_qemu/snap1.mem,snapshot=external --diskspec vda,file=/root/nested_qemu/L1_system.snap1.qcow2,snapshot=external --atomic

  Revert (3 steps — VM must be shut off first):                                                                                       
  # a. stop VM if running                                                                                                             
  sudo virsh destroy l1-ubuntu
                                                                                                                                      
  # b. reset overlay to empty (discards all writes since snapshot)                                                                    
  sudo rm /root/nested_qemu/L1_system.snap1.qcow2                                                                                     
  sudo qemu-img create -f qcow2 -b /root/nested_qemu/L1_system.qcow2 -F qcow2 /root/nested_qemu/L1_system.snap1.qcow2                                                                                           

  # c. restore memory state
  sudo virsh restore /root/nested_qemu/snap1.mem

  Key difference: virsh save/restore is a single round-trip (save stops the VM). External snapshots stay live — the VM keeps running
  after the snapshot — but reverting is manual.

[Trouble shooting for loading QM5 VM on L1 QEMU VM]

the QM5/chipsim VM was failing inside the L1 nested VM with cannot execute binary /usr/bin/qemu-system-x86_64: Permission denied, even with --privileged --security-opt apparmor=unconfined.                                        
                                                                                                                                                                                                                                            
  Root cause: AppArmor on the outer host (where libvirt runs your L1 VM, not L1 itself) matches profiles by binary path. The chipsim container's libvirtd binary lives at /usr/sbin/libvirtd — same path as the outer host's libvirtd binary
   — so the kernel auto-applied the host's libvirtd AppArmor profile to it across the container boundary, regardless of the apparmor=unconfined flag. That profile didn't whitelist Rocky's /usr/bin/qemu-system-x86_64, so every libvirt
  exec of QEMU got EACCES.                                                                                                                                                                                                                  
                  
  Fix (on the outer host — where you run virsh):                                                                                                                                                                                            
  
  sudo mkdir -p /etc/apparmor.d/disable                                                                                                                                                                                                     
  sudo ln -sf /etc/apparmor.d/usr.sbin.libvirtd /etc/apparmor.d/disable/usr.sbin.libvirtd
  sudo ln -sf /etc/apparmor.d/usr.lib.libvirt.virt-aa-helper /etc/apparmor.d/disable/usr.lib.libvirt.virt-aa-helper                                                                                                                         
  sudo apparmor_parser -R /etc/apparmor.d/usr.sbin.libvirtd                                                                                                                                                                                 
  sudo apparmor_parser -R /etc/apparmor.d/usr.lib.libvirt.virt-aa-helper                                                                                                                                                                    
                                                                                                                                                                                                                                            
  This persists across reboots (the symlinks in /etc/apparmor.d/disable/ tell AppArmor not to reload them). After this, the chipsim launcher progressed end-to-end



● For the specific "can't load QM5 VM" failure (cannot execute binary qemu-system-x86_64: Permission denied), yes — disabling the host's libvirtd AppArmor profile was the only fix.
